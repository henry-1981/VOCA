# BrideVOCA v0.2: Known/Unknown Quadrant Analysis

> Based on v0.2 Spec (40항목), Design, CLAUDE.md + 프로젝트 탐색.
> Designed under the constraint that "배포는 나중에. 아이들이 맥북 dev 서버에서 바로 사용. 40개 항목 점진적 완성".

---

## Current State Diagnosis

- **배포 압박 없음**: 아이들이 맥북에서 `npm run dev`로 사용 중. PWA/Vercel 배포는 후순위.
- **실사용 중**: 다온/지온이 개발 중간에 계속 하고 싶어함 → 앱이 이미 동작하고 있고, 아이들의 관심이 검증됨.
- **핵심 전환**: "배포가 안 될까봐" → "아이들이 쓰는 동안 깨지면 안 됨"으로 최대 리스크 이동.
- **테스트 기반 부재**: scaffold 1개뿐. dev 서버 사용 중 regression 방지가 중요.
- **What to stop doing**: 배포 인프라 작업(PWA manifest, Vercel, 오프라인 대응)에 시간 쓰기. 아이들이 체감하는 기능/폴리시에 집중.

## 2. Known Unknowns: Design Experiments (25%)

> 답이 없는 질문들. 각각 최소 실험을 설계.

### KU1. dev 서버 사용 중 regression을 어떻게 방지할 것인가?

**Diagnosis**: 아이들이 실사용하는 동안 기능 추가/수정이 계속됨. 테스트 없이 코드를 바꾸면 "어제 되던 게 오늘 안 됨" 발생 가능. 현재 테스트가 scaffold 1개뿐이라 안전망이 없음.

**Experiment**:
| Item | Detail |
|------|--------|
| Format | **테스트 피라미드 우선 구축**: Unit(핵심 로직) → Component(주요 화면) → E2E(학습 루프 1개) |
| Success criteria | (1) `npm test` 0 failures (2) E2E로 Day 1 Learn→Test 완주 검증 (3) lint+typecheck+build clean |
| Deadline | 기능 추가 시작 전 |
| Effort | ~4-5시간 (Claude 작성, HB 검증) |

**Promotion condition**: 테스트 통과 + 2회 연속 기능 추가 시 regression 없음
**Kill condition**: E2E 환경 세팅에 3시간 이상 소요 시 → Unit + Component만으로 축소

- **아이들 체감 순서**가 새 기준: 게이미피케이션 > 폴리시 > 안정화 > 배포

**Experiment**:

| Success criteria | 아이들이 "이거 생겼어!" 하고 반응하는 항목부터 완료 |
| Deadline | 점진적 (주말 단위) |

**Promotion condition**: 40개 중 30개 이상 완료

### KU3. 아이들이 동시에 같은 맥북 dev 서버를 쓸 때 프로필 충돌 문제는?

**Diagnosis**: dev 서버는 하나인데 iPad 2대에서 동시 접속. 현재 디바이스 바인딩은 구현되어 있으나, dev 환경에서의 동시 사용은 미검증:
- 같은 `localhost:3000`에 두 iPad가 접속하면?
- 디바이스 ID 구분이 localhost에서도 정상 동작하는가?
- Firestore 실시간 동기화가 dev에서도 되는가?

**Experiment**:
| Item | Detail |
|------|--------|
| Format | iPad 2대로 동시 접속 테스트 |
| Success criteria | 각 iPad에서 다온/지온 별도 데이터 표시, 간섭 없음 |
| Deadline | 아이들에게 주기 전 |
| Effort | 30분 (수동 테스트) |

**Promotion condition**: 간섭 없이 동작 → KK
**Kill condition**: 충돌 발생 → 포트 분리(3000/3001) 또는 시간대 분리로 대응

## 4. Unknown Unknowns: Set Up Antennas (5%)

> 예측 불가. 탐지 속도로 관리.

| # | Risk/Opportunity | Detection Method | Response Principle |
|---|-----------------|-----------------|-------------------|
| 1 | **아이들 피드백의 예상 밖 요구** | 사용 중 "이것도 해줘" 요청이 spec 밖일 수 있음 | spec 외 요청은 기록 후 v0.3에서 판단 |

| 5 | **게이미피케이션 과잉 자극** | 아이들 반응 관찰: 이펙트에 집중 vs 학습에 집중 | 절제 원칙 적용, 과하면 줄이기 |

## Core Principles

1. **아이들이 기준**: 다온/지온이 "와!" 하는 기능부터. 인프라는 아이들에게 투명.
2. **dev 서버 = 프로덕션**: 맥북에서 돌리는 게 실사용이므로, dev에서 깨지면 안 됨. 테스트 먼저.
3. **점진적 완성**: 주말 마감 없음. 매 세션마다 1-3개 항목씩 완성하고 아이들 반응 확인.
4. **관찰이 최고의 테스트**: 아이들이 쓰는 모습을 보면 스펙에 없는 인사이트가 나온다. 이걸 기록.
5. **게이미피케이션 절제 유지**: 이펙트 300-700ms, Review는 최소. 재미 > 과잉 자극.
